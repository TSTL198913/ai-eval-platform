## 二、Layer 1: 评分解析策略层（生产代码实现）

### 2.1 策略模式设计

> **生产代码位置**: [score_parsing.py](file:///d:/workspace/ai-eval-platform-refactor/src/domain/evaluators/strategies/score_parsing.py)

```python
@dataclass
class ParsedScore:
    """解析后的评分结果（含置信度）"""
    score: float
    confidence: float
    strategy: str
    sample_size: int = 1
```

### 2.2 策略链顺序

| 优先级 | 策略类 | 功能 | 置信度 |
|--------|--------|------|--------|
| 1 | LevelParseStrategy | 等级解析（1-5分制） | 0.90-0.95 |
| 2 | NumericExtractStrategy | 数字提取（0-1/百分制） | 0.90-0.95 |
| 3 | SemanticMappingStrategy | 语义映射（高/中/低风险） | 0.70-0.95 |
| 4 | KeywordFallbackStrategy | 关键词降级（积极/消极） | 0.30-0.70 |

### 2.3 语义映射表

#### 2.3.1 风险等级映射（金融/安全场景）

| 语义描述 | 映射分数 | 置信度 | 典型场景 |
|---------|---------|--------|---------|
| 高风险 / 高危 / 严重 | 0.1 ~ 0.3 | 0.90-0.95 | 安全漏洞、欺诈交易 |
| 较高风险 / 中高风险 | 0.3 ~ 0.5 | 0.85 | 可疑行为 |
| 中等风险 / 中风险 | 0.5 ~ 0.6 | 0.80 | 需要关注 |
| 较低风险 / 中低风险 | 0.6 ~ 0.8 | 0.85 | 基本安全 |
| 低风险 / 安全 / 无风险 | 0.8 ~ 1.0 | 0.90-0.95 | 正常可信 |

#### 2.3.2 质量等级映射（通用内容场景）

| 语义描述 | 映射分数 | 置信度 | 对应等级 |
|---------|---------|--------|---------|
| 优秀 / 完美 / 极佳 | 0.9 ~ 1.0 | 0.95 | 5分 |
| 良好 / 不错 / 较好 | 0.75 ~ 0.89 | 0.85-0.90 | 4分 |
| 一般 / 普通 / 中等 | 0.6 ~ 0.74 | 0.75 | 3分 |
| 较差 / 不好 / 差 | 0.4 ~ 0.59 | 0.75-0.80 | 2分 |
| 很差 / 糟糕 / 极差 | 0.0 ~ 0.39 | 0.90-0.95 | 1分 |

#### 2.3.3 相关性映射

| 语义描述 | 映射分数 | 置信度 |
|---------|---------|--------|
| 完全相关 / 切题 / 直击要点 | 0.9 ~ 1.0 | 0.90-0.95 |
| 基本相关 / 大部分相关 | 0.7 ~ 0.89 | 0.80 |
| 部分相关 / 有些跑题 | 0.4 ~ 0.69 | 0.70 |
| 不相关 / 答非所问 | 0.0 ~ 0.39 | 0.90-0.95 |

### 2.4 置信区间计算

```python
# 95% 置信区间计算
margin_of_error = 1.96 * sqrt(confidence * (1 - confidence) / sample_size)
ci_lower = max(0.0, score - margin_of_error)
ci_upper = min(1.0, score + margin_of_error)
```

**输出格式示例**: `score=0.85, CI=[0.78, 0.92], confidence=0.90, sample_size=1`

### 4.1 智能客服场景 Rubrics

#### 4.1.1 评分标准（1-5 分制）

| 分数 | 标准描述 | 关键检查点 |
|------|---------|-----------|
| **5分（优秀）** | 准确解决用户问题，语气亲和，引导至相关业务流程 | ✓ 问题解决<br>✓ 语气友好<br>✓ 业务推荐<br>✓ 无安全风险 |
| **4分（良好）** | 正确回答问题，但缺少业务推荐话术 | ✓ 问题解决<br>✓ 语气正常<br>✗ 业务推荐 |
| **3分（一般）** | 回答正确但语气生硬，信息不够完整 | ✓ 基本正确<br>✗ 语气生硬<br>✗ 部分遗漏 |
| **2分（较差）** | 回答部分错误，或未解决核心问题 | ✗ 部分错误<br>✗ 核心问题未解决 |
| **1分（很差）** | 答非所问，或包含虚假/有害信息 | ✗ 答非所问<br>✗ 虚假信息<br>✗ 安全风险 |

#### 4.1.2 System Prompt 模板

```
你是一个专业的智能客服质量评估专家。请根据以下标准对客服回答进行评分：

评分维度：
1. 问题解决度 (40%)：是否准确解决了用户的问题
2. 服务态度 (25%)：语气是否亲和、专业
3. 业务引导 (20%)：是否适当推荐相关业务或解决方案
4. 安全性 (15%)：是否包含错误信息或不当言论

评分等级：
5分 - 优秀：准确解决问题，语气亲和，有业务引导
4分 - 良好：正确回答，但缺少业务推荐
3分 - 一般：回答正确但语气生硬，信息不完整
2分 - 较差：部分错误，未解决核心问题
1分 - 很差：答非所问，或有虚假/有害信息

请输出：
- 总分（1-5）
- 各维度得分
- 详细理由
- 改进建议
```

## 五、Layer 4: 运营协议层（2026工业级新增）

### 5.1 多裁判集成架构

#### 5.1.1 架构设计

```
                    ┌───────────────────┐
                    │   最终评分决策      │
                    │  (多数投票/Borda)   │
                    └─────────┬─────────┘
                              │
          ┌───────────────────┼───────────────────┐
          │                   │                   │
    ┌─────▼─────┐      ┌─────▼─────┐      ┌──────▼──────┐
    │  裁判 A    │      │  裁判 B    │      │   裁判 C    │
    │ (GPT-4o)  │      │(Claude 3.5)│      │ (DeepSeek)  │
    └─────┬─────┘      └─────┬─────┘      └──────┬──────┘
          │                   │                   │
          └───────────────────┼───────────────────┘
                              │
                    ┌─────────▼─────────┐
                    │   输入样本        │
                    └───────────────────┘
```

#### 5.1.2 决策机制

| 机制 | 适用场景 | 计算公式 |
|------|---------|---------|
| 多数投票 | 二分类/等级判定 | 票数 > 2/3 则通过 |
| Borda 计数 | 多维度评分 | 排名加权求和 |
| 加权平均 | 连续值评分 | 按模型置信度加权 |
| 仲裁机制 | 争议样本 | 引入更高等级模型 |

#### 5.1.3 最低要求

- **最小裁判数**: 3 个
- **一致性阈值**: Cohen's Kappa ≥ 0.8
- **仲裁触发**: 意见分歧 > 40%

### 5.2 评判器对抗评估

#### 5.2.1 攻击类型与检测方法

| 攻击类型 | 描述 | 检测方法 | 防御措施 |
|---------|------|---------|---------|
| **位置偏差** | 对列表中靠前/靠后的选项偏好 | AB 测试、打乱顺序 | 随机化选项顺序 |
| **谄媚攻击** | 模型对自身输出打分偏高 | 盲测（隐藏来源） | 去标识化输入 |
| **Prompt 注入** | 通过指令注入影响评分 | 指令注入测试集 | 指令防护层 |
| **锚定效应** | 被参考答案影响 | 不提供参考答案测试 | 隐藏参考答案 |
| **长度偏见** | 偏好长回答 | 控制回答长度 | 标准化输入长度 |

#### 5.2.2 JudgeRobustnessEvaluator

**评估指标**:

| 指标 | 定义 | 安全阈值 |
|------|------|---------|
| 位置偏差率 | 不同位置的评分差异 | < 5% |
| 谄媚率 | 自身输出评分提升幅度 | < 10% |
| 注入成功率 | 恶意指令影响评分比例 | < 2% |
| 一致性退化率 | 对抗样本下 Kappa 下降 | < 15% |

### 5.3 标注者一致性度量

#### 5.3.1 Cohen's Kappa（双标注者）

```
Kappa = (Po - Pe) / (1 - Pe)

Po: 观测一致率
Pe: 期望一致率（随机概率）

阈值标准:
- Kappa ≥ 0.8: 优秀一致性
- 0.61-0.79: 良好一致性
- 0.41-0.60: 中等一致性
- ≤ 0.40: 一致性不足，需重新校准
```

#### 5.3.2 Fleiss' Kappa（多标注者）

```
适用于 3+ 标注者场景，计算方法类似但考虑多标注者分布

阈值标准:
- Kappa ≥ 0.75: 优秀一致性
- 0.40-0.74: 可接受一致性
- ≤ 0.39: 一致性不足
```

#### 5.3.3 Krippendorff's Alpha（任意度量类型）

```
适用于名义、顺序、区间、比率等多种数据类型
推荐用于语义评分的区间型数据

阈值标准:
- Alpha ≥ 0.80: 可靠性高
- 0.667-0.79: 可靠性可接受
- ≤ 0.666: 需要改进
```

### 5.4 Rubric 版本控制协议

#### 5.4.1 版本标识

```
{rubric_type}-v{major}.{minor}.{patch}-{hash}

示例:

- code_review-v2.1.0-b7d4e9
```

#### 5.4.2 变更日志规范

| 字段 | 说明 | 示例 |
|------|------|------|
| version | 版本号 | v2.1.0 |
| date | 变更日期 | 2026-06-29 |

#### 5.4.3 回滚机制

```
1. 保存每个版本的完整 Rubric 配置
2. 记录每个版本的校准数据（Golden Dataset 评分）
3. 当新版本导致评分漂移 > 15% 时自动触发回滚
4. 回滚后保留新版本数据用于分析
```

## 六、LLM-as-a-Judge 实施指南（2026升级）

### 6.1 裁判模型选择策略

| 层级 | 推荐模型 | 适用场景 | 成本 | 准确率 |
|------|---------|---------|------|--------|
| L1 初审 | 轻量模型 (DeepSeek-V3) | 批量初筛、低风险场景 | 低 | ~80% |
| L2 复审 | 中等模型 (GPT-4o-mini) | 常规评测、中风险场景 | 中 | ~90% |
| L3 终审 | 强模型 (GPT-4o / Claude 3.5) | 高风险场景、争议仲裁 | 高 | ~95% |

### 6.2 校准流程

```
1. 准备 Golden Dataset（专家标注的标准答案集，≥100样本）
2. 使用多裁判集成对 Golden Dataset 批量评分
3. 计算多裁判与专家标注的偏差和 Kappa 值
4. 如果偏差 > 5% 或 Kappa < 0.8，优化 Prompt 或更换模型
5. 通过校准后，输出置信区间报告用于生产评测
```

### 6.3 漂移监控

- **检测频率**：每周一次
- **检测方法**：使用 DriftEvaluator 对比历史评分分布
- **漂移阈值**：评分分布偏移 > 10% 触发告警
- **应对措施**：重新校准 + 检查 Prompt/Rubric 变更
