### 2.1 P0 缺陷技术评估

#### BUG-001: 并发数据库 Session 问题

**技术根因确认** ✅

查看 [session.py#get_db_session](file:///d:/workspace/ai-eval-platform-refactor/src/infra/db/session.py#L498-L538)：

```python
@contextmanager
def get_db_session() -> Generator:
    # 设计上是每次调用创建新 session，本身是线程安全的
```

**实际问题定位**：问题不在 `get_db_session()` 本身，而在于**测试代码中 Repository 实例被多线程共享**：

```
# test_concurrent_safety.py 中的问题模式
self.repo = EvaluationRepository(session)  # 单例

def writer(self):
    result = self.repo.save(...)  # 多线程共用 repo 和 session
```

**架构层面的修复方案对比**：

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **A. 测试侧修复** - 每个线程独立 session | 不改动业务代码，风险低 | 未从根源解决设计问题 | ⭐⭐⭐ |
| **B. Repository 层加锁** - ThreadLocal Session | 业务代码改动小 | 增加复杂度，可能影响性能 | ⭐⭐⭐⭐ |
| **C. UoW 模式重构** - Unit of Work 统一管理 | 架构清晰，彻底解决 | 改动量大，周期长 | ⭐⭐ |

**架构师建议**：
- **短期**：采用方案 A（测试侧修复），保证测试通过
- **中期**：采用方案 B，引入 `ScopedSession` 或 ThreadLocal 模式
- **长期**：规划方案 C，作为架构演进的一部分

**架构风险等级**：🟡 中风险
- 生产环境中如果存在共享 Session 的使用模式，风险升级为 🔴

#### BUG-002: SSL 上下文线程安全

**技术根因确认** ✅

查看 [llm_factory.py#create_llm_client](file:///d:/workspace/ai-eval-platform-refactor/src/domain/models/llm_factory.py#L219-L262)：

```python
# 好消息：代码中已经有缓存锁！
if use_cache:
    with _cache_lock:  # ✅ 已有锁保护缓存操作
        # 双重检查 + 创建客户端
```

**问题定位**：`_cache_lock` 只保护了缓存读写，没有保护 `_create_new_client()` 内部的 SSL 上下文创建。在 Windows 平台上，`ssl.create_default_context()` 不是线程安全的。

**架构层面修复方案**：

```python
# 方案1: 扩大锁范围（简单直接）
_llm_client_create_lock = threading.Lock()

def _create_new_client(provider: str, config: dict) -> BaseLLMClient:
    with _llm_client_create_lock:  # 全局锁保护客户端创建
        client_class = ModelRegistry.get_client_class(provider)
        return client_class(config)

# 方案2: 延迟初始化 SSL 上下文（更优雅）
# 在 httpx Client 中通过 SSLContext 单例模式
```

**架构师建议**：
- 采用方案1，简单可靠，影响范围可控
- 生产环境中 LLM Client 本就应该是单例，缓存机制已覆盖大部分场景
- 此问题在生产环境触发概率较低（客户端创建后即缓存），但测试环境频繁创建易触发

**架构风险等级**：🟢 低风险
- 生产环境有缓存保护，仅首次创建时可能触发
- 仅 Windows 平台受影响，Linux 平台正常

### 2.2 P1 功能评估

#### 语义评分解析

**架构影响分析**：

涉及 9 个评估器的评分解析逻辑（[base.py](file:///d:/workspace/ai-eval-platform-refactor/src/domain/evaluators/base.py) 基类）：

```
general.py, translation.py, semantic.py, qa.py,
factuality_evaluator.py, text.py, sentiment.py, classification.py
```

**架构设计建议**：

```python
# 建议在 BaseEvaluator 中实现统一的评分解析策略
class ScoreParser:
    """可配置的评分解析器"""

    def __init__(self, strategies: list[ScoreParseStrategy]):
        self.strategies = strategies

    def parse(self, text: str) -> float | None:
        for strategy in self.strategies:
            result = strategy.try_parse(text)
            if result is not None:
                return result
        return None

# 策略模式：数字提取、语义映射、正则匹配、默认值降级
```

**架构风险等级**：🟡 中风险
- 影响多个评估器的评分准确性
- 需定义语义评分标准，避免歧义

## 五、优化建议

### 5.1 架构师建议

1. **引入线程安全检查工具**
   - 使用 `threading-sanitizer` 或类似工具提前发现并发问题
   - 在 CI 中加入并发专项测试

2. **建立 Session 管理规范**
   - 明确 "一个请求一个 Session" 的原则
   - 禁止跨线程共享 Session

3. **配置变更测试框架化**
   - 建立配置变更测试基类
   - 所有可配置模块自动继承配置变更测试

### 5.2 项目经理建议

1. **每日站会同步进度**
   - 时间：每天早上 09:30
   - 内容：昨日完成、今日计划、阻塞问题

2. **缺陷跟踪机制**
   - 使用 BUG_REPORT.md 作为主跟踪文档
   - 每个缺陷指定唯一负责人
   - 每日更新状态

3. **质量门禁**
   - P0 修复后必须通过全量测试
   - P1 完成后必须有代码评审
   - 性能基线建立后纳入 CI 流程
