# AI 评测平台 - 性能基线与 SLA 体系

> 文档版本: v1.0
> 生成时间: 2026-06-29
> 设计理念: 先数据基线兜底，再业务目标迭代（漏斗模型）
> 方法论来源: Gemini 性能基线阈值体系

---

## 一、总体策略

### 1.1 漏斗模型

```
┌──────────────────────────────────────────────────────┐
│  第二阶段：业务期望 SLA（向上对齐）                     │
│  目标：满足业务场景要求                                 │
│  输入：产品需求、用户体验目标                            │
│  输出：分级阈值体系（红线/预警/期望）                    │
├──────────────────────────────────────────────────────┤
│  第一阶段：测试数据基线（向下摸底）                     │
│  目标：了解系统物理极限                                 │
│  输入：基准压力测试、资源监控                            │
│  输出：各组件性能基线数据、瓶颈定位                       │
└──────────────────────────────────────────────────────┘
```

### 1.2 核心原则

1. **数据驱动**：不拍脑袋，所有阈值基于实测数据
2. **分级管理**：红线 / 预警 / 期望 三级阈值
3. **持续迭代**：基线随版本更新，阈值随业务调整
4. **多维评估**：延迟 + 吞吐量 + 错误率 + 资源利用率

## 二、第一阶段：测试数据基线（摸底）

### 2.1 基线测量方法

#### 2.1.1 测量维度

| 维度 | 指标 | 单位 | 说明 |
|------|------|------|------|
| **延迟** | P50 / P95 / P99 | ms | 百分位延迟，关注长尾 |
| **吞吐量** | QPS / TPS | req/s | 每秒处理请求数 |
| **错误率** | Error Rate | % | 失败请求占比 |
| **资源** | CPU / Memory / IO | % / MB | 资源利用率 |

#### 2.1.2 测量规范

```
测量流程:
1. 环境准备（固定硬件/软件配置）
2. 预热阶段（确保缓存/JIT生效）
3. 基准测量（多轮迭代取稳定值）
4. 数据清洗（排除异常值）
5. 统计分析（计算百分位数、标准差）
```

#### 2.3.2 瓶颈定位方法

```
瓶颈排查顺序:
1. 下游 LLM API（最常见瓶颈）
   → 检查: 令牌速率、并发限制、响应时间

2. 数据库连接池
   → 检查: 活跃连接数、等待队列、慢查询

3. 异步任务队列
   → 检查: 队列长度、消费速率、堆积情况

4. 应用服务器资源
   → 检查: CPU、内存、GC、线程数

5. 网络带宽
   → 检查: 出入带宽、延迟、丢包
```

## 三、第二阶段：业务期望 SLA（分级阈值）

### 3.1 三级阈值体系

```
┌─────────────────────────────────────────────────┐
│  🔴 红线阈值（SLA 崩塌线）                        │
│  超过此值 = 系统不可用 / 严重故障                  │
│  触发: P0 告警 + 自动降级/熔断                    │
├─────────────────────────────────────────────────┤
│  🟡 预警阈值（容量规划线）                        │
│  达到此值 = 需要扩容 / 优化                       │
│  触发: P2 告警 + 容量规划                         │
├─────────────────────────────────────────────────┤
│  🟢 业务期望值（体验目标）                        │
│  正常运行 = 用户体验良好                          │
│  目标: 95% 时间在此区间内                         │
└─────────────────────────────────────────────────┘
```

### 3.2 具体阈值定义

#### 3.2.1 延迟阈值

| 场景 | 🟢 期望值 | 🟡 预警值 | 🔴 红线值 |
|------|----------|----------|----------|
| 单次同步评估 | < 500ms | < 1s | > 3s |
| 单次异步评估 | < 300ms | < 800ms | > 2s |
| API 端到端响应 | < 800ms | < 1.5s | > 5s |
| 批量评测（100条） | < 30s | < 60s | > 120s |

#### 3.2.2 吞吐量阈值

| 场景 | 🟢 期望值 | 🟡 预警值 | 🔴 红线值 |
|------|----------|----------|----------|
| 单实例 QPS | ≥ 50 | < 30 | < 10 |
| 集群总 QPS | ≥ 500 | < 300 | < 100 |
| 任务队列处理速率 | ≥ 100/s | < 50/s | < 20/s |

#### 3.2.3 错误率阈值

| 指标 | 🟢 期望值 | 🟡 预警值 | 🔴 红线值 |
|------|----------|----------|----------|
| 总错误率 | < 0.1% | > 0.5% | > 1% |
| 超时错误率 | < 0.05% | > 0.2% | > 0.5% |
| 5xx 错误率 | < 0.01% | > 0.1% | > 0.5% |
| 4xx 错误率（业务错误） | < 1% | > 5% | > 10% |

#### 3.2.4 资源利用率阈值

| 资源 | 🟢 期望值 | 🟡 预警值 | 🔴 红线值 |
|------|----------|----------|----------|
| CPU 利用率 | < 60% | > 80% | > 95% |
| 内存利用率 | < 70% | > 85% | > 95% |
| 数据库连接池使用率 | < 50% | > 80% | > 95% |
| 任务队列堆积 | < 100 | > 1000 | > 10000 |

## 四、CI/CD 性能门禁

### 4.1 门禁规则

```
代码合并准入:
├── 单元性能测试
│   ├── ✅ 核心组件 P95 延迟不退化 > 20%
│   ├── ✅ 吞吐量不下降 > 15%
│   └── ✅ 无新增性能异常
│
├── 集成性能测试（夜间执行）
│   ├── ✅ API 端点 P95 延迟正常
│   ├── ✅ 错误率 < 0.1%
│   └── ✅ 资源利用率正常
│
└── 全链路压测（每周执行）
    ├── ✅ 满足当前 SLA 阈值
    ├── ✅ 瓶颈无恶化
    └── ✅ 容量余量 > 30%
```

### 4.2 性能退化判定

```
退化等级:
├── 轻微退化（5-10%）
│   ├── 记录但不阻塞
│   └── 纳入优化 backlog
│
├── 中度退化（10-20%）
│   ├── 发出警告
│   └── 要求下次迭代前优化
│
└── 严重退化（> 20%）
    ├── 阻塞合并
    └── 必须修复后才能上线
```

### 4.3 基线更新机制

| 触发时机 | 更新方式 | 审批要求 |
|---------|---------|---------|
| 架构重大变更 | 重新全面测量 | 架构师审批 |
| 核心组件升级 | 增量更新基线 | 技术负责人审批 |
| 性能优化完成 | 更新基线值 | 测试负责人审批 |
| 常规迭代（季度） | 校准微调 | 自动更新 + 人工确认 |

## 六、项目亮点提炼（简历/面试用）

> **"我设计了漏斗式性能基线体系，先通过数据摸底掌握系统物理极限，再结合业务目标定义三级 SLA 阈值，实现了从'拍脑袋'到'数据驱动'的性能管理升级。"**

具体可展开：

1. **漏斗模型**：先基线摸底（客观极限），再业务对齐（主观目标）
2. **三级阈值**：红线（故障）/ 预警（扩容）/ 期望（体验）分层管理
3. **多维评估**：延迟 + 吞吐量 + 错误率 + 资源利用率 四维指标
4. **CI 门禁**：性能退化 > 20% 自动阻塞合并，防止性能劣化
5. **全链路覆盖**：单元 → 组件 → 全链路 三级压测体系
