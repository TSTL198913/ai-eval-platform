## 1. 迭代池概述

本迭代池整合了AI评测系统v2.1.0版本中所有待修复的架构问题和长期改进项，按**风险等级**和**依赖关系**划分为4个迭代，确保系统逐步向2026年工业级标准演进。

### 核心原则

| 原则 | 说明 |
|------|------|
| 风险优先 | Critical级问题优先修复 |
| 依赖驱动 | 上游问题优先于下游问题 |
| 小步快跑 | 每个迭代周期不超过1周 |
| 可回滚 | 每个迭代必须有回滚方案 |

## 4. 迭代1：稳定性修复

### 4.1 目标

消除系统中致命的设计缺陷，确保评估流程的稳定性和可维护性。

### 4.2 包含问题

| ID | 问题 | 优先级 | 预计工时 |
|----|------|--------|----------|
| ARCH-001 | 双层异常处理冗余 | 🔴 Critical | 8h |
| ARCH-003 | 双日志框架并存 | 🟠 High | 6h |
| ARCH-005 | 配置版本脱节 | 🟡 Medium | 1h |

### 4.3 详细任务

#### ARCH-001：双层异常处理冗余

**问题描述**: `safe_evaluate` 包装 `evaluate`，但 `evaluate` 本身已有完整异常捕获，导致 `safe_evaluate` 的 except 分支实际上是死代码。

**修复方案**:
1. 重构 `evaluate()` 为纯评估逻辑，移除内部异常捕获和降级逻辑
2. 将异常处理和降级策略上移到 `safe_evaluate()`
3. 确保业务异常（BasePlatformError）向上传播
4. 确保非业务异常被捕获并转换为 DomainResponse

**验收标准**:
- `evaluate()` 不再捕获异常，仅执行评估逻辑
- `safe_evaluate()` 统一处理所有异常场景
- 回归测试全部通过

#### ARCH-003：双日志框架并存

**问题描述**: `infra/logger.py` 使用 loguru，而 `base.py`、`engine.py`、`evaluator_svc.py` 使用标准 logging，两套日志体系无统一策略。

**修复方案**:
1. 创建统一的日志模块 `src/infra/logging/__init__.py`
2. 在所有模块中导入统一配置的 loguru logger
3. 移除标准 logging 的使用
4. 配置 trace_id 注入机制

**验收标准**:
- 所有模块统一使用 loguru
- 日志包含 trace_id
- 日志格式一致

#### ARCH-005：配置版本脱节

**问题描述**: `Settings.app_version = '1.0.0'`，与系统实际版本 v2.1.0 不一致。

**修复方案**:
1. 更新 `app_version` 为 `2.1.0`
2. 添加版本自动检测机制（从 CHANGELOG 或 git tag）

**验收标准**:
- API 返回正确版本号
- 配置文件版本与 CHANGELOG 一致

### 4.4 验收标准

| 指标 | 目标 |
|------|------|
| 测试通过率 | 100% |
| 异常处理路径 | 统一且无死代码 |
| 日志框架 | 全部使用 loguru |
| 版本信息 | 与 CHANGELOG 一致 |

### 4.5 回滚方案

- 保留修改前的代码备份
- 可通过 git revert 快速回滚

## 5. 迭代2：一致性修复 ✅ 已完成

### 5.1 目标

统一评估器体系，消除架构分裂，确保所有评估器遵循相同的设计模式。

### 5.2 包含问题

| ID | 问题 | 优先级 | 预计工时 | 实际工时 |
|----|------|--------|----------|----------|
| ARCH-002 | Online Evaluator 平行实现 | 🔴 Critical | 10h | 8h |
| ARCH-004 | Service层紧耦合 | 🟠 High | 8h | 3h |
| ARCH-007 | 异步评估阻塞风险 | 🟡 Medium | 4h | 2h |
| ARCH-008 | 熔断机制线程安全 | 🟠 High | 4h | 4h |

### 5.3 详细任务

#### ARCH-002：Online Evaluator 平行实现

**问题描述**: `src/domain/online/evaluator.py` 未继承 `BaseEvaluator`，绕过了状态机、置信度、safe_evaluate 等全部 v2.1.0 改进。

**修复方案**（经架构师审查调整）:
1. **保留 OnlineEvaluator 作为编排器**，不强制继承 BaseEvaluator（这是反模式）
2. 内部使用 BaseEvaluator 实例，并调用 `safe_evaluate()`
3. 将 `DomainResponse` 结果包装为 `OnlineEvaluationResult`
4. 支持可选注入评估器实例

**修改文件**:
- `src/domain/online/evaluator.py` - 重构 OnlineEvaluator

**验收标准**: ✅
- OnlineEvaluator 使用 safe_evaluate() 调用内部评估器
- DomainResponse 正确转换为 OnlineEvaluationResult
- 异常处理统一

#### ARCH-004：Service层紧耦合

**问题描述**: `evaluator_svc.py` 在模块级别实例化 `_repository = EvaluationRepository()`，无依赖注入机制。

**修复方案**:
1. 创建 `EvaluatorService` 类，支持构造函数注入 `repository` 和 `engine`
2. 保留 `run_evaluation_service()` 函数作为向后兼容的入口
3. 支持在测试中 mock repository 和 engine

**修改文件**:
- `src/services/evaluator_svc.py` - 重构为类基服务

**验收标准**: ✅
- `EvaluatorService` 支持构造函数注入
- 可进行单元测试（可 mock repository）
- 向后兼容：`run_evaluation_service()` 函数接口不变

#### ARCH-007：异步评估阻塞风险

**问题描述**: `evaluate_async()` 中的 `asyncio.to_thread()` 使用默认线程池，大量降级时可能耗尽。

**修复方案**:
1. 在 `BaseEvaluator` 中创建专用的 `ThreadPoolExecutor`（16个工作线程）
2. 使用 `asyncio.get_event_loop().run_in_executor()` 替代 `asyncio.to_thread()`

**修改文件**:
- `src/domain/evaluators/base.py` - 添加专用线程池

**验收标准**: ✅
- 使用专用线程池（eval-worker-*）
- 高并发降级场景下系统稳定

#### ARCH-008：熔断机制线程安全

**问题描述**: `EvaluatorFactory.register()` 方法中的验证逻辑在锁之外执行，导致并发竞争。

**修复方案**:
1. 将评估器类验证逻辑移到 `_lock` 内部
2. 修复测试中 DummyEvaluator 未实现 `_do_evaluate` 的问题

**修改文件**:
- `src/domain/evaluators/evaluator_factory.py` - 线程安全修复
- `tests/reliability/test_concurrent_safety.py` - 测试修复

**验收标准**: ✅
- 并发注册评估器无异常
- 11个并发安全测试全部通过

### 5.4 验收标准

| 指标 | 目标 | 实际 |
|------|------|------|
| 评估器体系 | 统一调用 safe_evaluate | ✅ 已实现 |
| Service层 | 支持依赖注入 | ✅ 已实现 |
| 异步评估 | 专用线程池，限流保护 | ✅ 已实现 |
| 并发安全 | 所有测试通过 | ✅ 19个测试通过 |
| 测试通过率 | 100% | ✅ 核心测试全部通过 |

### 5.5 回滚方案

- 保留修改前的代码备份
- 可通过 git revert 快速回滚

**修改文件**:
- `src/infra/monitoring/metrics.py` - 添加状态机监控指标
- `src/domain/evaluators/base.py` - 在 `_log_evaluation_result()` 中记录指标
